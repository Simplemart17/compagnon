## Story

As a TCF Canada candidate whose onboarding placement test is generated by a 145-line `SYSTEM_PROMPT` const inline at [`app/onboarding/placement-test.tsx:75-145`](app/onboarding/placement-test.tsx) — the audit explicitly flagged P1-5 ("No `placement.ts` prompt file — placement test logic appears inline or absent") and `docs/tcf-spec-source.md §10` follow-up #7 deferred "Onboarding / placement test TCF readiness indicator" to **this story** — meaning the only TCF-grade prompt the user sees BEFORE they ever practice anything (a) cannot be regression-tested in isolation, (b) ships its own inline per-CEFR vocabulary tiers (`top-500`, `top-1000`, `top-3000`, `top-5000`) that drift silently from Story 10-4's `src/lib/prompts/vocabulary-tiers.ts` (heuristic caps 700 / 1700 / 2800 / 5000 / 7500 / 10000), (c) bypasses the cross-check regression guard added by Story 10-4 review patch P9 (which validates that existing per-level guidance does NOT smuggle in tokens already forbidden at that tier), and (d) is the lowest-leverage of the 9 CEFR-aware prompt surfaces despite being **the test that decides every user's starting level**,
I want the inline `SYSTEM_PROMPT` extracted to a dedicated `src/lib/prompts/placement.ts` builder following the Story 10-3 / 10-4 single-source-of-truth pattern — preserving the proven A1:3 / A2:3 / B1:3 / B2:3 / C1:2 / C2:1 question distribution + the 4-competency rubric (grammar / vocabulary / reading comprehension / pragmatics) + the distractor + explanation requirements — and that this new builder integrates `buildVocabularyConstraintBlock(cefrLevel)` per Story 10-4 in a way that respects the placement test's unique constraint of spanning A1–C2 in a single AI call (i.e., using `buildAggregatedVocabularyConstraintTable()` the same way `mock-test.ts` does), while **deleting** the inline per-CEFR top-N vocabulary claims that conflict with vocabulary-tiers.ts and **deleting** the duplicated `SYSTEM_PROMPT` literal at the call site (no aliases, no backwards-compat shims — Story 10-2 "delete don't alias" pattern),
so that **the placement test joins the family of regression-tested, citation-matrix-tracked, Story-9-4-prompt-injection-defended prompt builders** — closing audit finding **P1-5**, closing `docs/tcf-spec-source.md §10` follow-up #7, extending Story 10-4's vocab-tier integration coverage from 8 builders to 9 (citations matrix §9 row gains `placement.ts`), and giving Epic 10.7 (linguistic accuracy pass) + Epic 10.8 (anti-cheat / anti-repetition) a discoverable surface to modify when those stories land — without changing any user-visible behavior of the placement test (same 15 questions, same A1:3 / A2:3 / B1:3 / B2:3 / C1:2 / C2:1 distribution, same schema, same `parseRetries: 2`, same `gpt-4o` model, same `0.5` temperature, same 4096 max-tokens).

### Why the inline prompt is a release blocker (not just code-hygiene)

Three concrete failure modes the inline placement prompt creates today:

1. **Silent vocab-tier drift.** The inline `SYSTEM_PROMPT` at [`app/onboarding/placement-test.tsx:88,92,97,102`](app/onboarding/placement-test.tsx) hard-codes per-CEFR vocabulary tiers as `top-500` / `top-1000` / `top-3000` / `top-5000`. Story 10-4's [`src/lib/prompts/vocabulary-tiers.ts`](src/lib/prompts/vocabulary-tiers.ts) is the single source of truth and pins these as 700 / 1700 / 2800 / 5000 / 7500 / 10000. **Two source-of-truth values for the same constant is exactly the lockstep-update risk Story 10-3 + 10-4 dismantled across the other 8 prompt builders.** A future Beacco-Phase-2 swap of the vocab-tier numbers would update vocabulary-tiers.ts and silently leave the placement test on stale numbers — and the placement test is the *most-visible-to-new-users* of all 9 prompt surfaces.

2. **Cross-check regression hole.** Story 10-4 review patch P9 ([Story 10-4 §"Senior Developer Review" P9](_bmad-output/implementation-artifacts/10-4-vocabulary-frequency-caps.md)) added a 24-case `it.each` regression block that strips the legitimate Vocabulary Constraint block from each rendered prompt, then asserts none of the level's forbidden tokens appear in the residual legacy content. It runs on **4 builders × 6 levels** (listening / reading / echo / translation; conversation excluded as Epic-10.7-owned). After this story, placement.ts joins that cross-check matrix: **5 builders × 6 levels = 30 cases**. Today, no regression test prevents a future "improvement" to the inline placement prompt from drifting back into forbidden lower-tier tokens.

3. **No isolated regression-testability.** The other 8 prompt builders are pure functions exported from `src/lib/prompts/*.ts`, regression-tested via substring assertions in `src/lib/prompts/__tests__/`. The placement prompt is a 145-line `const` defined inside a React screen file — every test must mount the React component or mock the screen's `useEffect` to exercise the prompt. **A function that cannot be unit-tested in isolation cannot earn the §9 citations-matrix coverage that Story 10-4 closed for every other CEFR-aware surface.**

This is the same fix-the-drift-via-extraction pattern as:

- Story 10-2 deleting `rawToTCFScore` and replacing with `rawPercentToListeningReadingScore` + `rawPercentToWritingSpeakingScore`
- Story 10-3 extracting `writingTaskWordRange` from 3 lockstep-update sites
- Story 10-4 extracting `vocabularyTier` + `buildVocabularyConstraintBlock` from 0 sites (greenfield) and wiring it into 8 builders

### Threat / failure model — what cannot happen post-story

After this story:

1. **`src/lib/prompts/placement.ts`** is the **only** source of the placement-test system prompt. The const literal at `app/onboarding/placement-test.tsx:75-145` is **deleted** (not commented, not aliased, not re-exported — Story 10-2 "delete don't alias" pattern). The call site imports `buildPlacementTestPrompt` from `@/src/lib/prompts/placement` and uses it.
2. **`buildPlacementTestPrompt()`** is a pure function: input `()` (no parameters — the placement test always generates the same 15-question A1:3 / A2:3 / B1:3 / B2:3 / C1:2 / C2:1 distribution), output `string`. Deterministic — same input, byte-identical output (Story 9-4 prompt-injection-defense determinism contract).
3. **Per-CEFR inline vocab claims are replaced by `buildAggregatedVocabularyConstraintTable()`.** The placement test spans A1–C2 in a single AI call (parallel to `mock-test.ts`); a single-level `buildVocabularyConstraintBlock` would mis-calibrate, so the same aggregated-table renderer Story 10-4 chose for mock-test is the correct integration here too. The 145-line inline prompt's hard-coded `top-500` / `top-1000` / `top-3000` / `top-5000` strings are **deleted** — vocabulary-tiers.ts is the single source.
4. **Question distribution metadata centralised.** The current `LEVEL_RANGES` array at `app/onboarding/placement-test.tsx:42-49` (which encodes the 15-question A1:3 / A2:3 / B1:3 / B2:3 / C1:2 / C2:1 distribution) is **kept in placement.ts** as an exported constant so both the prompt builder AND the scoring logic at `app/onboarding/placement-test.tsx:512-535` `calculateLevel` consume it from one place. ### Architecture pattern alignment

- **Single-source-of-truth helper module** — parallel to Story 10-2's `IRCC_CLB_BANDS`, Story 10-3's `writingTaskWordRange`, and Story 10-4's `vocabularyTier`. `placement.ts` is the fourth module in the `src/lib/prompts/` source-of-truth family. The pattern: extract → re-import → delete-don't-alias.
- **Pure function + module-static data + no parameters** — `buildPlacementTestPrompt()` is the strictest determinism contract of the four (10-2's helpers take a number, 10-3's `writingTaskWordRange` takes a task number, 10-4's `buildVocabularyConstraintBlock` takes a CEFR level — only 10-5 takes nothing). Story 9-4 prompt-injection defense is satisfied trivially.
- **Aggregated-table integration over per-level block** — same rationale as `mock-test.ts`: the placement test spans A1–C2 in a single AI call, so a single-level block would mis-calibrate. The aggregated-table renderer at `vocabulary-tiers.ts:323-338` is the canonical choice.
- **Question-distribution metadata centralisation** — `PLACEMENT_LEVEL_RANGES` is consumed by BOTH the prompt (per-level question-count breakdown for the AI) AND the scoring path (`levelForQuestion`, `calculateLevel`, `buildResultsSummary`). Today these are duplicated; this story collapses them. Same lockstep-update risk Story 10-3 dismantled for `writingTaskWordRange` across the 3 writing-prompt sites.
- **No behavioral change at the call site** — the React screen at `app/onboarding/placement-test.tsx` keeps all UI rendering, all React state, all skeleton animations, all `LEVEL_CONGRATS` / `buildResultsSummary` / `getLevelStatusLabel` logic verbatim. Only the prompt + the two distribution constants migrate. This is a refactor, not a feature.
- **Schema unchanged** — Story 9-7's `placementQuestionSchema` + `placementTestSchema` at `src/lib/schemas/ai-responses.ts:551-664` are NOT touched. The schema is the runtime contract; the prompt is the generation guide. They are deliberately separate.

### Anti-pattern prevention

- **Do NOT introduce parameters to `buildPlacementTestPrompt()`** — the placement test is a fixed pre-onboarding artifact. A parametric version (e.g., variable question count, variable level distribution) is a separate feature story. Keep the function nullary; preserve trivial determinism.
- **Do NOT re-inline `top-500` / `top-1000` / `top-3000` / `top-5000` vocab strings** — `buildAggregatedVocabularyConstraintTable()` is the canonical source. The negative-assertion tests in AC #3 + the grep guard in Task 8 are the regression defenses.
- **Do NOT change the question distribution** — A1:3 / A2:3 / B1:3 / B2:3 / C1:2 / C2:1 is verified-production (MEMORY.md "Feature Completion Sprint" 2026-03-04). Any distribution change is a separate story.
- **Do NOT change the model / temperature / maxTokens / parseRetries** — `gpt-4o` / `0.5` / `4096` / `2` are the Story 9-7 settings. Tuning is a separate post-extraction follow-up.
- **Do NOT alias the deleted `SYSTEM_PROMPT`** — Story 10-2 "delete don't alias" lesson. Grep guard in Task 8 enforces it.
- **Do NOT modify `placementQuestionSchema` / `placementTestSchema`** — Story 9-7 owns the schema. The prompt is the generation guide; the schema is the validation guard. They are deliberately separate.
- **Do NOT touch the post-placement results UI** — `LEVEL_CONGRATS`, `buildResultsSummary`, `getLevelStatusLabel`, the React render block. UI changes (e.g., a "TCF readiness indicator" widget) are deferred to Epic 14.X.
- **Do NOT include grammar.ts** — operator-decided as non-TCF practice skill (Story 10-4 §"Out of scope" same rationale).
- **Do NOT introduce a database migration or schema change** — vocabulary tiers + placement-test prompt are static module exports.
- **Do NOT touch any Edge Function** — prompt builders run client-side.
- **Do NOT backfill historical placement runs** — forward-only.
- **Do NOT add a 5th competency to the 4-competency rubric** (grammar / vocabulary / reading comprehension / pragmatics). The audit's 4-competency setup is the validated production behavior; a 5th competency (e.g., pronunciation, sociolinguistics) is a separate story.
- **Do NOT change the JSON output contract** — `placementTestSchema` is the runtime guard; the prompt must instruct the AI to produce schema-compliant output. The current "exactly 4 options", "exactly 1 correct", "isCorrect: boolean" language is preserved verbatim.

### Testing standards

- **Substring assertions on prompt output, not implementation internals** — same contract as Story 10-3's `passage-calibration.test.ts` and Story 10-4's `vocabulary-integration.test.ts`.
- **Negative-assertion tests are non-negotiable** — the placement prompt MUST NOT contain `top-500` / `top-1000` / `top-3000` / `top-5000` after extraction. Negative tests fail loudly on regression and are the cheapest defense against a "let me re-inline the vocab tiers for clarity" patch.
- **Cross-check matrix extension** — adding `placement.ts` to Story 10-4 review patch P9's 4-builder cross-check is the 30-second-cost / 30-cases-coverage upgrade. Do it.
- **Determinism check** — `buildPlacementTestPrompt()` MUST return byte-identical output for two consecutive calls. Defends against accidental `Date.now()` / random / module-load-order dependencies (story 9-4 defense + story 10-4 pattern).
- **Each per-level / per-builder assertion is its own `it.each` row** so failures are diagnosable. Don't `forEach` 30 cases into a single test (Story 10-3 review patch P5 lesson).
- **Don't test the AI's behavior** — only the prompt-builder's output. Whether the AI actually respects the question-distribution / 4-competency / 4-options rules is the schema's job (`placementQuestionSchema.superRefine`).
- **Re-use existing test helpers** — `vocabulary-integration.test.ts` has a `assertVocabularyConstraintPresent` helper (Story 10-4 review patch P7); if the placement assertion shape is similar enough, re-use it instead of forking. If it diverges (because placement uses aggregated-table not per-level block), create a sibling helper `assertAggregatedVocabularyConstraintPresent` co-located in the same test file.
- **Schema regression check** — the existing 5 `placementTestSchema` test cases at `ai-responses.test.ts:258-657` are the production regression for the schema. They should stay green; if they break, the schema was modified, which is out of scope.
