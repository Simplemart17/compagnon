## Story

## Background — Why This Story Exists

### What the audit finding looks like in production

The audit row [`shippable-roadmap.md §1 P3-9`](_bmad-output/planning-artifacts/shippable-roadmap.md) reads:

> P3-9 | No item bank caching — every exercise regenerated by GPT (cost + no anti-repetition) | `src/hooks/use-exercise.ts`

And the Epic 10.8 deliverable line at [`shippable-roadmap.md` line 166](_bmad-output/planning-artifacts/shippable-roadmap.md):

> 10.8 Anti-cheat & frequency anti-repetition — add basic item dedupe (hash of question stem) to avoid showing the same generated MCQ across sessions.

The current `generateExercise` flow at [`src/hooks/use-exercise.ts:99-252`](src/hooks/use-exercise.ts) per skill:

```
listening:  buildListeningExercisePrompt(...) → chatCompletionJSON → result.questions[]
reading:    buildReadingExercisePrompt(...)   → chatCompletionJSON → result.questions[]
grammar:    buildGrammarExercisePrompt(...)   → chatCompletionJSON → result.questions[]
writing:    inline prompt-gen prompt          → chatCompletionJSON → { prompt }
```

None of these branches reads what the user has already seen. With `temperature: 0.4`, GPT-4o reliably collapses to its most-likely completion for a given (skill, CEFR-level) tuple — meaning the same question stems recur across sessions for the same user even though the audio script / passage / writing prompt may differ at the surface level. The "anti-cheat" framing in the audit deliverable line is the candidate-facing manifestation (a user who memorized the answer to "Selon le texte, pourquoi…" from session 3 needs no comprehension to answer it in session 5); the "frequency anti-repetition" framing is the cost / UX manifestation (the user feels the app is broken because every drill looks the same).

### What 9-8's 3-day-bucket gives us (verified-correct, NOT touched by 10-8)

Story 9-8 (2026-05-09) shipped a deterministic 3-day-bucket topic selector at [`src/lib/prompts/speaking.ts`](src/lib/prompts/speaking.ts) `computeTopicBucket` for the Expression Orale tasks — same userId + same 3-day window → same picked topic, so retakes within the bucket see a consistent prompt and the user can't shop for a topic by retrying. The Story 10-8 design **does NOT touch** the speaking-task selector: it is a different domain (TCF mock-test speaking section), has a different anti-cheat model (publisher-graded production task vs. self-graded MCQ practice), and the deterministic-bucket approach would be hostile to the 4 practice skills where users WANT fresh content per session. Story 10-8 adds anti-repetition to the 4 practice-skill drills; speaking's per-bucket determinism is correct and stays.

### 1. Extract `hashText` to a shared `src/lib/text-hash.ts` module

- [x] **CREATE** [`src/lib/text-hash.ts`](src/lib/text-hash.ts) exporting:
  ```typescript
  /**
   * djb2 hash over Unicode code points, base-36-encoded.
   *
   * Iterating with `for...of` walks code points (not UTF-16 code
   * units), so emoji and surrogate-pair text hash cleanly. The
   * result is a short opaque string with no embedded free text —
   * safe to pass through Sentry breadcrumbs and to persist in DB
   * columns indexed for membership queries.
   *
   * Used by:
   *   - `src/lib/realtime-transcript.ts` `fallbackKey` (story 9-5)
   *   - `src/lib/exercise-dedup.ts` `hashQuestionStem` (story 10-8)
   */
  export function hashText(text: string): string {
    let hash = 5381;
    for (const ch of text) {
      hash = ((hash << 5) + hash + (ch.codePointAt(0) ?? 0)) >>> 0;
    }
    return hash.toString(36);
  }
  ```
- [x] **UPDATE** [`src/lib/realtime-transcript.ts:97-103`](src/lib/realtime-transcript.ts) — delete the private `hashText` function declaration; replace with `import { hashText } from "@/src/lib/text-hash"` at the top of the file. Story 10-2 "delete don't alias" pattern.
- [x] **VERIFY** Story 9-5's regression tests at `src/lib/__tests__/realtime-dedup.test.ts` stay green — the hash function output is byte-identical pre/post extraction (no logic change).
- [x] **CREATE** `src/lib/__tests__/text-hash.test.ts` — 6 cases: empty string → constant `5381.toString(36)`; single ASCII char (deterministic); Unicode emoji codepoint (NOT UTF-16 surrogates — `for...of` walks codepoints); idempotence (`hashText(x) === hashText(x)`); collision-resistance smoke (10 distinct short strings → 10 distinct hashes); pin the literal output for a sentinel string (regression guard).

### 2. Create the pure-helper module `src/lib/exercise-dedup.ts`

- [x] **CREATE** [`src/lib/exercise-dedup.ts`](src/lib/exercise-dedup.ts) exporting:

  ```typescript
  import { hashText } from "@/src/lib/text-hash";
  import type { GeneratedExercise } from "@/src/hooks/use-exercise";
  
  /**
   * Normalize a question stem for hash-based dedup.
   *
   * (a) NFC-normalize so visually-identical strings with different
   *     code-point compositions hash to the same value (per Story 9-4
   *     `sanitizeMemoryContent` precedent).
   * (b) lowercase via `toLocaleLowerCase("fr")` so French diacritics
   *     fold predictably (e.g., `È` → `è`, not Turkish dotted-i drift).
   * (c) trim leading/trailing whitespace.
   * (d) collapse all interior whitespace (incl. NBSP / tab / newline)
   *     to a single ASCII space.
   *
   * The output is a stable canonical form fed to `hashText`.
   * Determinism: two callers passing the same input get the same
   * output. Idempotence: `normalize(normalize(x)) === normalize(x)`.
   */
  export function normalizeQuestionStem(text: string): string {
    return text
      .normalize("NFC")
      .toLocaleLowerCase("fr")
      .trim()
      .replace(/\s+/gu, " ");
  }
  
  /** djb2 hash over the normalized stem. Short opaque base-36 string. */
  export function hashQuestionStem(text: string): string {
    return hashText(normalizeQuestionStem(text));
  }
  
  /**
   * Extract question-stem hashes from a generated exercise across all
   * 4 skill shapes. Returns an empty array for unknown skills or for
   * exercises missing the expected shape (defensive — never throws).
   */
  export function extractExerciseHashes(exercise: GeneratedExercise): string[] {
    if (exercise.skill === "writing") {
      const promptText = exercise.writingPrompt?.prompt ?? "";
      return promptText ? [hashQuestionStem(promptText)] : [];
    }
    // listening / reading / grammar share the same MCQ question[] shape
    return exercise.questions
      .map((q) => q.question ?? "")
      .filter((s) => s.length > 0)
      .map(hashQuestionStem);
  }
  
  /**
   * Filter MCQ questions, dropping any whose normalized stem hash
   * appears in the seen set. Operates on the schema-conformant
   * question array AFTER Zod validation (Story 9-7); never sees
   * partial / malformed data.
   */
  export function filterUnseenQuestions<T extends { question: string }>(
    generated: T[],
    seen: ReadonlySet<string>
  ): T[] {
    if (seen.size === 0) return generated;
    return generated.filter((q) => !seen.has(hashQuestionStem(q.question)));
  }
  
  /**
   * Test a single writing prompt for repetition. Returns true if the
   * prompt is in `seen` (and should trigger a regenerate).
   */
  export function isWritingPromptSeen(prompt: string, seen: ReadonlySet<string>): boolean {
    if (seen.size === 0 || prompt.length === 0) return false;
    return seen.has(hashQuestionStem(prompt));
  }
  ```

- [x] **All functions are pure** — no I/O, no Date / Math.random / clock reads, no Supabase calls. Regression-testable without mocks.
- [x] **`normalizeQuestionStem` is idempotent** — `normalize(normalize(x)) === normalize(x)` for any string.
- [x] **`hashQuestionStem` is deterministic** — same input always produces the same output across runs.

**Given** two semantically-identical question stems that differ only in case + leading whitespace + NFC composition (e.g., `"  Selon le Texte, Pourquoi…"` vs `"selon le texte, pourquoi…"`)
**When** `hashQuestionStem` is called on both
**Then** both produce the same hash output

**Given** an exercise with `skill: "writing"` and a 200-char `writingPrompt.prompt`
**When** `extractExerciseHashes(exercise)` is called
**Then** the result is a 1-element array with the prompt's normalized hash

**Given** an exercise with `skill: "listening"` and `questions: [{ question: "Q1" }, { question: "" }, { question: "Q2" }]`
**When** `extractExerciseHashes(exercise)` is called
**Then** the result is a 2-element array (empty stems filtered out)

### 3. Create the Supabase query helper `src/lib/exercise-dedup-db.ts`

- [x] **CREATE** [`src/lib/exercise-dedup-db.ts`](src/lib/exercise-dedup-db.ts) exporting:

  ```typescript
  import { supabase } from "@/src/lib/supabase";
  import { captureError } from "@/src/lib/sentry";
  import type { CEFRLevel, TCFSkill } from "@/src/types/cefr";
  
  /** Default look-back window — the last N completed exercises matching (skill, cefr_level). */
  export const DEFAULT_SEEN_LIMIT = 100;
  
  /**
   * Fetch the question-stem hashes the user has seen across their
   * last N completed exercises at the given (skill, cefr_level).
   *
   * Returns an empty set on Supabase error so generation proceeds
   * unfiltered — never block the user on a dedup query failure
   * (Story 9-10 auth-cache resilience pattern). Sentry breadcrumb
   * fires via `captureError(_, "exercise-dedup-fetch")`.
   *
   * The query uses indexed columns (user_id + skill + cefr_level) +
   * an order by completed_at (covered by the existing per-user RLS
   * scan); the new GIN index on `question_stem_hashes` is NOT
   * needed for this query — it accelerates inverse-direction
   * "find exercises containing hash X" queries (none currently exist,
   * but reserved for Epic 13.x server-side item bank caching).
   */
  export async function getSeenHashes(
    userId: string,
    skill: TCFSkill,
    cefrLevel: CEFRLevel,
    opts: { limit?: number } = {}
  ): Promise<Set<string>> {
    const limit = opts.limit ?? DEFAULT_SEEN_LIMIT;
    try {
      const { data, error } = await supabase
        .from("exercises")
        .select("question_stem_hashes")
        .eq("user_id", userId)
        .eq("skill", skill)
        .eq("cefr_level", cefrLevel)
        .eq("completed", true)
        .order("completed_at", { ascending: false })
        .limit(limit);
      if (error) {
        captureError(error, "exercise-dedup-fetch", {
          skill,
          cefrLevel,
        });
        return new Set();
      }
      const seen = new Set<string>();
      for (const row of data ?? []) {
        const hashes = (row as { question_stem_hashes?: string[] | null }).question_stem_hashes;
        if (Array.isArray(hashes)) {
          for (const h of hashes) seen.add(h);
        }
      }
      return seen;
    } catch (err) {
      captureError(err, "exercise-dedup-fetch", {
        skill,
        cefrLevel,
      });
      return new Set();
    }
  }
  ```

- [x] **Sentry telemetry contract** — the `captureError` call uses Story 9-3's allowlist-safe extras (only `skill` and `cefrLevel`, both small enum strings; no userId, no raw query result). Verified by re-reading `src/lib/sentry.ts` `SENTRY_EXTRAS_ALLOWLIST`.

### Architecture pattern alignment

- **Pure-helpers + thin I/O layer** — `exercise-dedup.ts` is pure; `exercise-dedup-db.ts` isolates Supabase I/O. Mirrors Story 9-2 `evaluatePromotion`'s pure-helper pattern (deterministic regression-testable core) + Story 9-10's resilience-on-DB-failure pattern (empty result instead of throw).
- **Story 10-2 "delete don't alias" pattern** — the private `hashText` in `realtime-transcript.ts` is deleted, not re-exported as a legacy alias. Future drift between two djb2 implementations is impossible.
- **Forward-only schema growth (Story 10-2 / 10-6 / 9-8 precedent)** — pre-10-8 `exercises` rows have NULL `question_stem_hashes`; the dedup helper treats NULL as "no hashes." No backfill. Same pattern Story 10-6 used for `mock_tests.section_scores.speaking.task{1,2,3}` 4-dim → 5-dim growth.
- **Retry-once contract (Story 9-7 precedent)** — `chatCompletionJSON` already retries once on schema-parse failure. Story 10-8 layers a second retry-once on top — dedup-driven, NOT schema-driven. Two retries are still bounded; the worst-case latency is 2× the AI call time and is logged via Sentry. Tag the retry call with `feature: "exercise-<skill>-retry"` so Sentry distinguishes retry-on-dedup from retry-on-parse.
- **Sentry telemetry contract (Story 9-3 precedent)** — all `captureError` extras are allowlist-safe (small enum strings or bounded integers; no userId, no question text, no AI response payload). Verified by re-reading `src/lib/sentry.ts` `SENTRY_EXTRAS_ALLOWLIST`.

### Anti-pattern prevention

- **Do NOT introduce a server-side item bank cache** — that's Epic 13.x. Story 10-8 is per-user, client-driven dedupe.
- **Do NOT share item banks across users** — privacy concern; Epic 13.x with operator opt-in.
- **Do NOT touch `src/lib/prompts/speaking.ts` `computeTopicBucket`** — speaking-task dedup is the deterministic 3-day-bucket contract (Story 9-8).
- **Do NOT add a runtime fallback that re-hashes** — `normalizeQuestionStem` is idempotent; calling `hashQuestionStem(hashQuestionStem(x))` is a smell.
- **Do NOT inline the djb2 hash in `exercise-dedup.ts`** — import from `text-hash.ts` (Story 10-2 "single source of truth").
- **Do NOT block the user on a Supabase query failure** — `getSeenHashes` returns an empty set on error; generation proceeds.
- **Do NOT retry more than once** — two retries (first call + one retry) is the budget. A third retry is over-engineering and doubles the worst-case cost.
- **Do NOT escalate the retry temperature beyond `0.6`** — `1.0+` produces low-quality MCQ distractors and risks Zod parse failures (Story 9-7 retry path interaction).
- **Do NOT throw on retry-exhausted** — accept the dup + breadcrumb. Users want exercise content, not an error screen because the AI ran out of diversity.
- **Do NOT add a UI indicator** ("3/5 questions are fresh") — Epic 14.x UI scope; ship dedup invisibly first.
- **Do NOT backfill pre-10-8 rows** — forward-only.
- **Do NOT touch the RLS policy** — the existing per-user policy covers the new column.
- **Do NOT use a JSONB column for hashes** — Postgres array + GIN index is the right primitive for membership queries.

### Testing standards

- **Pure-helper tests use literal sentinel values for the hash output** — `expect(hashQuestionStem("Selon le texte")).toBe("<literal-base-36>")` so a future accidental hash-algorithm change fails the test, not just the integration.
- **Mocked-Supabase tests use the `setupSupabaseMock` pattern** from `speaking-mock-test-persist.test.ts` Story 10-6 review patch P1 — exposes the insert mock so tests can inspect the payload (verify `question_stem_hashes` is populated).
- **Integration tests in `use-exercise.test.ts` mock BOTH `chatCompletionJSON` AND `getSeenHashes`** — each branch (empty seen / partial overlap / full overlap / retry-fresh / retry-exhausted) is its own `it()` case with explicit mock returns.
- **Determinism / idempotence cases are mandatory** — hash functions and normalization helpers must pin determinism explicitly.
- **Negative assertions defend deletion claims** — `realtime-transcript.ts` should NOT contain a local `function hashText(` definition after Task 1; regression test via grep at `src/lib/__tests__/text-hash.test.ts` (or co-located in `realtime-dedup.test.ts` as a smoke check).
