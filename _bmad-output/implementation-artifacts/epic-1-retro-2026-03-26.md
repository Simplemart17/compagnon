## Challenges

1. **Same bug classes repeated across 5+ stories:**
   - Hardcoded hex colors (Stories 1.2, 1.3, 1.4, 1.5, 1.7)
   - ActivityIndicator spinners instead of skeleton loaders (Stories 1.2, 1.3, 1.4, 1.5, 1.7)
   - Missing accessibility attributes (Stories 1.1, 1.2, 1.3, 1.4, 1.7)
   - Missing `captureError` in catch blocks (Stories 1.1, 1.2)
   - Missing error states with Retry + Back (Stories 1.3, 1.7)
   - **Root cause:** Original feature code was written before design conventions were established, and no automated enforcement existed

2. **NativeWind className limitation** — `className` attributes can't use `Colors.*` design tokens (requires inline `style`). Converting all className hex to inline styles is a significant refactor flagged as tech debt in Story 1.5.

3. **Critical logic bugs invisible behind correct UI** — UTC date bugs, question count mismatches, and PII leaks all required deep code inspection to find, not visual testing.

---

## Key Insights

1. **Verification epics are high-value** — catching 5 critical bugs before users see them justifies the investment
2. **Polish must ship with the feature, not after** — deferring design tokens, accessibility, and error handling to a cleanup story creates compounding debt
3. **Story-to-story intelligence is a process worth keeping** — documenting bug patterns prevents regressions
4. **Automated enforcement prevents recurring bug classes** — CI grep for hex colors and accessibility lint rules are one-time investments with permanent returns
5. **A cleanup epic before Epic 2 will establish guardrails** — CI enforcement, standardized ACs, and component architecture planning ensure future epics don't accumulate the same debt

## Significant Discovery: Cleanup Epic Required

During the retrospective, the team identified that recurring bug classes (hardcoded hex, spinners, missing accessibility) appeared in 5 of 7 stories because no automated enforcement existed. Rather than jumping into Epic 2 (which restructures the home screen), the Project Lead decided to insert a **cleanup epic** to:

- Establish CI guardrails that prevent recurring bug classes
- Resolve known tech debt (className hex conversion)
- Plan Epic 2's component architecture before writing code
- Standardize story ACs with polish requirements

**Impact on Epic 2:** Epic 2 is deferred until the cleanup epic completes. This is expected to result in faster, cleaner Epic 2 execution with fewer polish bugs.

**Recommendation:** Define the cleanup epic and create stories via SM agent's `create-story`.
