## What Went Well

### 1. The "bounded-budget cap" pattern reached 4 iterations and is now a durable idiom

Epic 12 brought the project's per-array memory-bound discipline to maturity:

| Story | Constant                          | Module                                  |
| ----- | --------------------------------- | --------------------------------------- |
| 9-4   | `MAX_MEMORY_CHARS = 300`          | `src/lib/memory.ts`                     |
| 11-7  | `MAX_PROMPT_MEMORIES = 3` etc.    | `src/lib/prompts/conversation.ts`       |
| 12-6  | `MAX_TRANSCRIPT_ENTRIES = 200`    | `src/lib/transcript-cap.ts`             |
| 12-12 | `MAX_PRONUNCIATION_HISTORY = 50`  | `src/lib/pronunciation-history.ts`      |

Each iteration follows the same structure: single `MAX_*` constant + pure helper FIFO append-with-cap + Jest drift detector pinning the constant + drift detector pinning the consumer's call site. Story 12-12 was the easiest of the four to ship because the pattern is now mechanical — the engineer could read 12-6's test file and produce 12-12's by analogy. **Codify this as a CLAUDE.md "Bounded-budget cap pattern" subsection** so future stories adopting the idiom inherit the structure without re-deriving it.

### 2. "Single chokepoint refactor" became the standard scope discipline

Six Epic 12 stories used a single-chokepoint refactor as their primary scope:

- 12-1: `RealtimeOrchestrator` class as the chokepoint for 14 realtime responsibilities
- 12-2: `bootstrapAuth()` module as the chokepoint for the auth listener (replacing 7 per-consumer listeners)
- 12-3: 4 RPC functions as the chokepoint for activity mutations
- 12-7: `cache.ts` fork as the chokepoint for encrypted vs plaintext storage (0 consumer changes)
- 12-9: `app/_layout.tsx` render-branch as the chokepoint for email-verification (0 changes to login/signup/forgot-password)
- 12-11: `parseUpstreamError()` as the chokepoint for upstream-error sanitization (5 callers updated)

Each chokepoint absorbed the change inside a small module surface, leaving 5–8 consumer files compiling unchanged. This is the dominant Epic 12 reason that 366 net tests landed with zero user-visible regressions.

### 3. The "delete don't alias" pattern held across the entire epic

Story 10-2 introduced "delete the old code; don't alias it via a backwards-compat shim." Epic 12 applied it 11 times — every story that replaced an existing surface either deleted the predecessor outright or moved it inside a private helper:

- 12-1: 1,354-line god-hook **deleted**, replaced by 116-line pure-consumer hook
- 12-2: 359-line `useAuth` listener body **deleted**, hook shrinks to 70 lines
- 12-3: 2 `TODO(epic-10-schema-hardening)` JSDoc markers **deleted** (debt paid)
- 12-3: unused `maxLevel` helper **deleted**
- 12-5: `ExpoPlayAudioStream.destroy()` call **deleted** (replaced by ref-counted release)
- 12-6: pre-12-6 `[...this.transcript, entry]` unbounded spread **deleted**
- 12-8: `password.length < 6` check **deleted**
- 12-9: implicit "no email verification" path **deleted** (now load-bearing gate)
- 12-10: legacy npm-audit-ignore behavior **deleted** (CI now fails on high)
- 12-11: pre-12-11 JSON-parsing leak branch **deleted**
- 12-12: pre-12-12 unbounded `[...prev.history, result]` spread **deleted**

Drift detectors enforce each deletion: every story's source-drift test file has a negative-guard regex that fails CI if the deleted pattern is re-introduced. **This is now the single most productive pattern in the project.**

### 4. The 3-layer adversarial review continued to find real bugs that quality gates miss

Across 12 stories, ~183 review patches landed. Patterns identical to Epic 10 + 11:

- **Blind Hunter** (no project context) — strongest on test-correctness (story 12-8 HIGH `captureError(*, password)` walker false-positive caught vacuous pass; story 12-5 H2 `isConnected` short-circuit caught dead-code-path).
- **Edge Case Hunter** (project read access) — strongest on cross-file invariants (story 12-3 ECH found stale `cefr_level` corruption path; story 12-9 M1 caught the doc-claim that "an unverified user shouldn't have reached profile-load" was partly false because Story 9-6's listener runs loadProfile BEFORE the gate render).
- **Acceptance Auditor** (spec + diff) — strongest on what-was-NOT-touched verification + missing-AC catches (story 12-2 AA: missing MFA test case; story 12-6 AA: missing AC #2 BDD #2 offline-queue ordering test).

The HIGH/MED/LOW triage discipline kept patch rounds bounded — **5–33 patches per story**, mostly settling in the 13–17 range.

### 5. Test growth scaled linearly with audit findings closed — and tests act as our load-bearing memory

+366 tests in one epic. The ratio (tests-per-audit-finding ≈ 30) means roughly one test per ~half-hour of original audit prose — which is the right calibration: every closed finding gets a regression detector that survives future refactors. Drift detectors specifically (reading source from disk + comment-stripping per Story 12-2 P12 + regex-asserting structural invariants) emerged as the single most productive defensive test family. **Epic 12 alone added ~14 new drift detector files.**

### 6. Operator runbooks became a first-class deliverable, not an afterthought

Six stories shipped operator runbooks alongside source code:

- 12-3: `runbooks/atomic-activity-rpcs.md`
- 12-8: `runbooks/auth-password-policy.md`
- 12-9: `runbooks/auth-email-verification.md`
- 12-10: `runbooks/dependency-vulnerability-policy.md`

This was new for Epic 12 — Epic 10 + 11 only shipped CLAUDE.md paragraphs. **The runbook pattern should be a checklist item for any future story whose closure depends on operator action.**

### 7. The Sentry telemetry allowlist contract (Story 9-3) held byte-perfectly across all 12 stories

Six Epic 12 stories added new `feature` tags (12-1: 7 new `persist-conversation-phase-a-*`; 12-4: 1 `orchestrator-session-null-on-event`; 12-5: 1 `audio-stream-release-when-zero`; 12-6: 1 `transcript-cap-evicted` + 2 new bounded-int extras; 12-7: 3 `secure-cache-*`; 12-9: 1 `email-verification-resend`). Each addition was triaged against `SENTRY_EXTRAS_ALLOWLIST` and added explicitly with a 1-line rationale. **Zero raw PII leaked into Sentry events across the entire epic.** Story 12-8 R2-P3's string-literal-aware balanced-paren walker for the `captureError(*, password)` negative guard is now reusable for any future PII-leak drift detector.

## What Didn't Go Well

### 1. Review-patch rounds scaled with story sprawl — and Story 12-8 needed TWO rounds

The Epic 10 retro flagged 8–14 patches per story as the steady state. Epic 12 settled around 13–21 — meaningfully higher. Story 12-8 needed TWO review rounds:

- Round 1: 19 patches (HIGH × 3 + MED × 6 + LOW × 10)
- Round 2: 14 more patches (HIGH × 3 + MED × 3 + LOW × 8)
- **Net: 33 review patches on one story** — the densest of any Epic 12 story.

The round-2 patches were all introduced BY round-1 patches (adversarial round-2 caught regressions in round-1 fixes themselves). The pattern matches the Epic 9/12 lesson: patches introduce new bugs while fixing old ones. **Future stories with > 15 round-1 patches should automatically trigger a round-2 review** rather than skipping to merge.

### 2. Story 12-11 review-round-1 patches were documented but NOT APPLIED

A novel decision this epic: the operator chose to merge Story 12-11 with the documented 14 patches (3 HIGH + 5 MED + 6 LOW) **not applied**. Rationale was workflow velocity — patches were enumerated but the operator chose to move directly to the next story. The deferred items are listed in `sprint-status.yaml` for 12-11.

Risk assessment:
- 12-11 is a single-file refactor of `parseUpstreamError` with 5 caller sites; the chokepoint reduces blast radius.
- Round-1 HIGH patches included tightening the categorical label allowlist enforcement and adding a Sentry breadcrumb on body-read timeout — both safety-net improvements, not correctness fixes.
- The 6-case `isRetryable` compatibility test pins the load-bearing client behavior (HTTP-status substring retry), so a future regression that re-leaks a model name would be caught by the drift detector, not by the deferred patches.

Still, **this sets a precedent that should be tracked.** If telemetry post-rollout surfaces an issue traceable to one of the deferred 14 patches, the precedent costs more than the velocity it bought. Recommend a follow-up sweep in Epic 15.x ("apply or formally reject the 14 deferred 12-11 patches") so the debt doesn't decay invisibly.

### 3. The Story 12-9 M1 finding: a CLAUDE.md doc-claim was partly false

Story 12-9 M1 (review-round-1) corrected a CLAUDE.md claim: "an unverified user shouldn't have reached profile-load" was **partly false** because Story 9-6's auth listener runs `loadProfile` BEFORE the gate renders. The gate prevents UI exposure but does NOT prevent the listener-driven cache read.

This is the first time in the project that a CLAUDE.md architecture paragraph has been retroactively flagged as load-bearing-but-wrong. Two follow-ups:

1. The read-side gap (loadProfile-before-verification) remains open — file as Epic 13/14 follow-up.
2. **CLAUDE.md paragraphs should describe the actual data flow, not the intended behavior.** Future paragraphs adding "X should not reach Y" claims need to be verified against the real call sequence before merge.

### 4. Story 12-3 pgTAP tests are still NOT CI-wired

Story 12-3 shipped 11 pgTAP-style assertions in `supabase/migrations/__tests__/atomic_activity_rpcs_test.sql` — including the load-bearing 100-concurrent-update tests that satisfy the Epic 12 AC. They are manual-run only (`psql -f`). The same applies to Story 11-6's `match_error_pattern_test.sql` and Story 11-4's `rate_limit_test.sql`. **Three migrations' worth of test coverage lives outside CI** — operator has to remember to run them manually.

Epic 15.3 is the planned home for CI-wiring these tests. Until then, the Epic 12 AC #2 (concurrent phone+web no-loss) is only verified architecturally (atomic `INSERT … ON CONFLICT … DO UPDATE` row-lock semantics) plus deterministically (the SQL test math is correct). Real concurrent-connection load testing (`pgbench -c 100 -j 100 -t 1`) is **not happening today**. That's acceptable but it's a real gap.

### 5. Story 12-1's parallelization had a documentation gap that took round-1 patches to close

Story 12-1's Phase A 6-way `Promise.allSettled` parallelization was correctly implemented but originally had NO tests proving the parallelization actually happened. Round-1 P2/P3 added "max-skew < 50ms across the 6 Phase A slots" assertion + "per-slot Sentry tag fires on rejection" assertion + "Phase B `checkCefrPromotion` fires AFTER all Phase A promises settle" assertion. **The implementation was correct; the spec asked for parallelization but the AC didn't explicitly require proof.**

This is structurally similar to Story 10-8 AA1 (missing integration test) and Story 12-6 AC #2 BDD #2 (missing offline-queue ordering test). Pattern: **performance/parallelism claims in AC need their own test case**, not just an implementation matching the spec.

### 6. Story 12-7's "one-shot migration" semantic took 4 round-1 patches to harden

Story 12-7 (encrypted profile cache) shipped with a migration path that read legacy plaintext AsyncStorage entries on first post-deploy read, wrote them to SecureStore, then deleted the legacy. Round-1 P1 caught that an **expired** legacy entry was being migrated to SecureStore with a fresh `Date.now()` timestamp (reviving stale data); P2 added a module-level `Map` to serialize concurrent migration races; P3 fixed an offline cold-launch path that hard-threw because `readSecureCacheIgnoreTTL` returned null pre-migration; P4 added envelope shape validation to all 3 reader helpers.

Each patch closed a real failure mode that the original implementation missed. The lesson: **one-shot migrations require explicit failure-mode enumeration during dev-story** (write the failure list BEFORE the code: "what happens if migration runs twice / runs while offline / runs against expired data / runs against malformed legacy data"). Add this to the bmad-dev-story checklist for any future migration story.

## Significant Discoveries

### 1. CLAUDE.md paragraphs can describe intended-not-actual behavior (Story 12-9 M1)

The first project-internal instance of a CLAUDE.md architecture paragraph being load-bearing-but-wrong. The gate-prevents-UI-but-not-listener-driven-read gap is filed as a future follow-up. **Action: every new CLAUDE.md "X prevents Y" claim must be verified against the real call sequence — not against the spec — before merge.**

### 2. Story 12-10 reframed the audit finding because the world moved underneath it

P1-13 originally cited "3 high (xmldom XML injection, recursion DoS)" — but by 2026-05-14 the Expo SDK 49 → 55 chain had silently dropped xmldom from the dep tree. The story shipped a CI gate against regression instead of a fix for a defect that no longer exists. **Audit findings have decay; the 2026-05-06 audit is now 8 calendar days old at Epic 12 close and at least some findings have moved.** Recommend a brief "audit refresh pass" before Epic 13 to flag which other 2026-05-06 findings may have similarly drifted.

### 3. The deferred 14 patches in Story 12-11 set a workflow precedent

First time in the project that documented review patches were merged without being applied. Risk is low (single-chokepoint refactor with drift detector pinning load-bearing behavior) but the precedent itself is worth tracking. **Recommend filing as "12-11-deferred-patch-sweep" in backlog with a Epic 15.x revisit window.**

### 4. The transcript cap (12-6) architecturally enables Epic 13.1 — but does NOT fix it

Story 12-6 caps `state.transcript` at 200 entries, making the FlatList virtualization budget bounded. But Epic 13.1's actual problem (FlatList re-render storm via `extraData={transcript.length}`) is NOT fixed by 12-6 — every `setState` still triggers an `extraData` invalidation. **12-6 reduces the WORK per render but not the FREQUENCY of renders.** Epic 13.1 still has the full original scope.
