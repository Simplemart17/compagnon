## What Went Well

### 1. The "publisher-anchored" + "delete don't alias" patterns made the spec-sourcing story (10-1) the load-bearing foundation

Story 10-1 produced four artifacts that every subsequent Epic 10 story consumed verbatim:

- `docs/tcf-spec-source.md` (11 sections) — the source-of-truth document with WebFetch-snapshot integrity headers.
- `docs/tcf-spec-citations.md` (9 → 11 sections) — the matrix that pins every TCF-derived value to a section anchor.
- `src/lib/__tests__/tcf-spec.test.ts` (12 cases) — fails CI if a new TCF claim is missing a matrix row.
- 4 source snapshots under `docs/tcf-canada-snapshots/` with retrieval timestamps + SHA-256.

This meant 10-2 through 10-8 never had to re-litigate "what does the publisher say"; they could read the matrix, find the ✗ DELTA row, and ship the fix with a citation. The Story 10-2 "delete don't alias" pattern (linear `rawToTCFScore` deleted + replaced with per-skill IRCC-anchored conversions) became a template the next six stories reused.

### 2. Three-layer adversarial review (Blind Hunter / Edge Case Hunter / Acceptance Auditor) consistently caught real bugs

Every Epic 10 story raised 20-30 review findings → averaged 8-14 patches per story. Patterns:

- **Blind Hunter** (no project context) — strongest at spotting test-correctness defects ("the assertion passes for the wrong reason") and surface-level contract violations.
- **Edge Case Hunter** (project read access) — strongest at cross-file invariants and path-traversal bugs (story 10-7 ECH1 caught a chat-bubble rendering regression that the implementer missed; story 10-8 ECH3 caught a seed-poisoning loop in the writing dedup pipeline).
- **Acceptance Auditor** (spec + diff) — strongest at "what the spec said NOT to touch is faithfully not touched" verification + missing-AC catches (story 10-8 AA1 caught the missing `use-exercise.test.ts` integration test).

The HIGH/MED/LOW triage discipline + "reject false positives explicitly with rationale" pattern kept review patches scoped and the patch round bounded.

### 3. Forward-only schema growth + "treat NULL as no data" became a reusable pattern

Three stories shipped forward-only schema or contract changes (10-2 publisher-scale total_score, 10-6 sociolinguistic 5th dimension JSONB, 10-8 question_stem_hashes column). Each story documented:

- The discontinuity (pre-X rows hold old shape; post-X rows hold new shape).
- The read-side defense (NULL / missing field treated as "no data").
- The cost-of-backfill decision (skipped in all 3 — operator-grade backfill is Epic 17.x scope per `docs/tcf-spec-source.md §10b`).

No production breakage from any of the 3 transitions. The pattern is now durable enough to write into a CLAUDE.md section ("Forward-only schema growth pattern").

### 4. Per-CEFR vocabulary tiers (Story 10-4) became a cross-cutting integration that 8 prompt builders consumed

Story 10-4 shipped `src/lib/prompts/vocabulary-tiers.ts` with `buildVocabularyConstraintBlock(cefrLevel)` + `buildAggregatedVocabularyConstraintTable()`. 10-5 integrated it into placement.ts; 10-6 verified it stayed wired into speaking.ts; 10-7 verified it across all 9 CEFR-aware builders. Adding a new prompt builder that doesn't pull from the vocab module is now caught by the `vocabulary-integration.test.ts` cross-check matrix (parameterized via `it.each` across 6 CEFR levels × N builders).

### 5. Story file as context-engine + Senior Developer Review section was institutionalized

Every Epic 10 story file has the structure: Story → Background → Threat/failure model → Acceptance Criteria → Tasks → Dev Notes (architecture pattern alignment + pulling-forward-lessons + source tree components + anti-pattern prevention + testing standards + references) → Dev Agent Record → Senior Developer Review (AI). The Senior Developer Review section captures triage decisions verbatim so the next story can read "story 10-X review-patch P3 closed this finding" and treat the prior cycle as durable institutional memory.

### 6. The Sentry telemetry allowlist contract (Story 9-3) absorbed Epic 10 cleanly

Three Epic 10 stories added new `captureError` extras keys (10-1 `feature` / `phase`, 10-6 schema-parse breadcrumbs, 10-8 `generatedCount` / `filteredCount` / `seenCount` / `retries`). Each addition was reviewed against `SENTRY_EXTRAS_ALLOWLIST` and added explicitly with a 1-line rationale. No raw PII leaked into Sentry events across the entire epic.

### 7. Cross-story NOT-touched discipline held

Every Epic 10 story file enumerates the surfaces it does NOT touch (typically 5-10 file paths with rationale). The Acceptance Auditor verifies the negative claims diff-by-diff. Result: zero unintended cross-story regressions — story 9-4 prompt-injection defense, story 9-5 voice transcript dedup, story 9-7 schema retry, story 9-8 speaking pipeline, and story 9-10 auth resilience all hold byte-identically across 8 Epic 10 stories.

## What Didn't Go Well

### 1. Review-patch rounds settled around 8-14 patches per story — the "70% done" lesson from Epic 9 still applies

Epic 9 retro called this out as a load-bearing pattern, and Epic 10 confirmed it: every story passed type-check + lint + tests + format BEFORE review, then absorbed 8-14 patches. The pattern is **not** a defect, but it is a real cost we now need to budget:

- 10-1: 12 patches
- 10-3: 8 patches
- 10-4: 9 patches
- 10-5: 14 patches (the densest review round — placement extraction had many test-assertion-tightening patches)
- 10-6: 6 patches (the leanest — sociolinguistic was a tightly-scoped schema-evolution story)
- 10-7: 10 patches
- 10-8: 9 patches

**Total Epic 10 review patches: 68+.** Quality gates catch ~70% of issues; the 3-layer review reliably finds the remaining 30%. Budget review-patch rounds as a first-class story phase, not a bug.

### 2. Stories 10-1 and 10-5 hit the same "test passes for the wrong reason" anti-pattern

Both stories had assertions like `expect(prompt).toContain("force est de constater")` that passed because `vocabulary-tiers.ts` legitimately surfaced the token elsewhere in the prompt — not because the assertion's target row contained it. 10-5 review patch P3 fixed it; 10-7 P5 reapplied the same fix at a different surface (writing.ts discourse-markers block); 10-8 P3 explicitly strengthened mocked-Supabase tests so the per-column `.eq()` filter calls fire with the right values.

**Lesson:** scope assertions to the smallest surface that proves the contract. Add a "negative-substring assertion" pattern review check to the Senior Developer Review template — does this test fail if a future patch silently inverts the wired-up contract?

### 3. The `_bmad*` gitignore footgun from Epic 9 stayed fixed but the pattern of "story file invisible to git" was almost re-discovered by 10-3

Story 10-3 was nearly merged with the story file untracked (caught by the dev agent's `git status` self-check). The Epic 9 retro lesson ("`git status` shows the story file as untracked-but-not-ignored") held — but only because every story-file template since 10-1 includes the self-check checkbox. Worth promoting to a pre-commit hook in a future hardening sub-story (Epic 16.X or a follow-up to story 9-9's deploy-substrate work).

### 4. Operator-action items in `docs/tcf-spec-source.md §10b` accumulated without a tracking surface

Epic 10 surfaced 5 operator-action items: 3 PDF downloads (Manuel du candidat, Expression Orale samples, Expression Écrite samples), 1 IRCC CLB table re-verification, 1 Beacco volume acquisition (~€120-180). All are documented in `§10b` but there's no ticket or reminder system — they live only in the source-of-truth doc. Two of them (Beacco for full vocab calibration; Manuel for 9-criterion speaking rubric) gate Phase-2 follow-up sub-stories that don't have story files yet.

**Lesson:** when a story surfaces an operator-action item, also create a "deferred" entry in `sprint-status.yaml` under a new `operator_actions:` section so the queue is visible in the same place as story status.

### 5. Migration deployment remains manual; no Epic 10 story actually deployed its migration

Story 10-8 added migration `20260511000000_exercise_question_stem_hashes.sql`. Per `CLAUDE.md` Deploy substrate ("SQL migrations stay manual until story 16-6 lands the rollback playbook"), the migration ships but does not auto-deploy. The migration ran on local dev but has not run against production Supabase. This is intentional (Story 16-6 owns the rollback playbook + auto-deploy) but worth flagging: every Epic-10-shipped story is in a forward-only-with-manual-deploy state until 16-6 closes.

### 6. Test count grew rapidly (322 → 894 = +572) but no tests were retired or consolidated

Test growth was deliberate (each story shipped its regression surface) but no story took the opportunity to consolidate overlapping tests. For example, the vocabulary-integration cross-check matrix grew from 4 builders (10-4) to 9 builders (10-7) without revisiting whether the 18-case `it.each(ALL_LEVELS × ALL_TASKS)` parametrization could be tightened. Test-suite runtime is still under 3 seconds total, so this is not yet a perf concern — but a Test Coverage & QA Infrastructure story (Epic 15.1) should review the surface.

### 7. The Story 10-7 vs 10-8 boundary was awkward — 10-7's connector-misclassification fix needed Story 10-4's vocab-tier infrastructure

Story 10-7 split the conversation.ts debate-mode "advanced connectors" list into 3 sub-categories AND fixed echoes in writing.ts + placement.ts. But the underlying mechanism (vocab-tier forbidden-lower-tier integration) lived in Story 10-4. This was a coordination success but a story-scope smell: 10-7's "small bug-fix story" pulled on infrastructure from a much larger predecessor. Worth noting that small linguistic-accuracy stories should NOT be sequenced before the infrastructure stories they depend on.

## Significant Discoveries (Affect Epic 11 / 13 / 17)

### D1: Anti-repetition is a per-skill contract, not a unified primitive

Story 10-8 shipped per-user hash-set dedupe for the 4 practice skills, but speaking-task dedup (Story 9-8 3-day-bucket) and Realtime conversation dedup (unique by construction) are governed by different contracts. A future unified item-bank cache (Epic 13.x) must respect these per-domain contracts; do NOT collapse them into one mechanism.

### D2: AI prompt-builder + Zod schema layer is now the validation primitive for ALL practice-skill content

Stories 10-3 / 10-4 / 10-5 / 10-6 / 10-7 all routed through `chatCompletionJSON(_, schema, _)` for content generation. Schema parse failures trigger Story 9-7's retry-once path; dedup runs AFTER parse (Story 10-8). The architectural ordering — Zod first, then dedup — is now a load-bearing invariant. Epic 11.1's `report_correction` tool-call work should preserve this ordering when it lands.

### D3: Per-CEFR + per-skill calibration tables are stable enough to expose as configuration

Listening (`buildListeningExercisePrompt`), reading (`buildReadingExercisePrompt`), writing (`writingTaskWordRange`), speaking (`TASK_DURATIONS` + topic libraries), and placement (`PLACEMENT_LEVEL_RANGES`) all have per-CEFR or per-task configuration tables that are publisher-anchored and unlikely to change without an explicit publisher-edition follow-up. Worth surfacing as a single "TCF Configuration" exports surface that the planner UI (Epic 14.x) and analytics dashboard (Epic 13.x) can read.

### D4: The shared djb2 hash extraction (Story 10-8) is the start of a "shared text-utils" namespace

Story 10-8 created `src/lib/text-hash.ts` to share djb2 between Story 9-5 voice transcript dedup + Story 10-8 exercise dedup. Future text utilities (normalization, sanitization, NFC handling) could live there too. The Story 9-4 `sanitizeMemoryContent` and Story 10-8 `normalizeQuestionStem` both do NFC + Unicode-aware operations independently — there's no near-term need to consolidate, but the namespace is ready when it's wanted.

### D5: Forward-only schema growth + GIN-index-deferred is the right Supabase migration pattern

Story 10-8 review patch P11 dropped a GIN index that was "reserved for Epic 13.x" because: (a) write amplification on every insert, (b) `CREATE INDEX` without `CONCURRENTLY` locks the table, (c) Supabase wraps migrations in a transaction which forbids `CONCURRENTLY`. The lesson: only ship the migration the current story needs; defer indexes (and any other write-amplifying construct) to the consumer story. Document the deferred-index pattern in Story 16-6's rollback playbook when it lands.

### D6: Placement test (Story 10-5) is a "fixed pre-onboarding artifact" pattern that other one-shot AI surfaces could adopt

`buildPlacementTestPrompt()` takes no parameters because the placement test is a fixed pre-onboarding artifact. The prompt output is deterministic (Story 9-4 contract). Other one-shot AI surfaces — e.g., the post-conversation feedback prompt (Story 4-3), the writing-task prompt generator (Story 10-3) — could adopt the same "deterministic-builder + no caller-supplied state" pattern to simplify their test surface and eliminate prompt-injection risk.
